# Настройка Notion для AI-дайджест бота

Бот синхронизирует профили проектов из Notion раз в неделю (по понедельникам, 6:00 UTC) и по запросу через `/sync`. Эта инструкция объясняет как подключить Notion и оформить страницы проектов так, чтобы парсер их понял.

## 1. Создать integration

1. Открой [notion.so/my-integrations](https://www.notion.so/my-integrations).
2. **«+ New integration»**.
3. Имя: `ai-digest-bot` (произвольное).
4. Workspace: тот, где живут страницы проектов.
5. Capabilities: достаточно `Read content`. Запись не нужна.
6. Submit → скопируй **Internal Integration Token** (начинается с `secret_…`).
7. Положи токен в `.env` как `NOTION_API_KEY=...` и в GitHub Secrets под тем же именем.

## 2. Дать integration доступ к страницам

Notion integration НЕ имеет доступа к страницам по умолчанию — это надо явно разрешить.

Для каждой страницы проекта:
1. Открой страницу в Notion.
2. Кнопка `•••` (правый верхний угол) → **«Add connections»**.
3. Найди свой integration `ai-digest-bot` → подключить.

Альтернатива: подключи integration к корневой странице, и она получит доступ ко всем подстраницам автоматически.

## 3. Узнать `notion_page_id`

URL Notion-страницы выглядит так:
```
https://www.notion.so/Workspace/TF-Market-1a2b3c4d5e6f7890abcdef1234567890
```

`notion_page_id` — это **последние 32 hex-символа** из URL (после последнего дефиса). В примере выше: `1a2b3c4d5e6f7890abcdef1234567890`. Дефисы внутри id (если они есть в более коротком URL вида `?p=…`) не нужны — парсер их сам уберёт.

## 4. Отдать `notion_page_id` боту

Два способа:

**А. Через Telegram** (рекомендуется):
В чате с ботом отправь:
```
set tf_market 1a2b3c4d5e6f7890abcdef1234567890
```
Формат: `set <slug> <notion_page_id>`. Slug — из `seed.py` (`tf_market`, `tf_clo`, `bage`, `vpn_bot`).

**Б. Прямо в Supabase Studio**: открыть таблицу `projects`, в строке проекта проставить колонку `notion_page_id`.

## 5. Оформить блок «Профиль для AI-бота» на странице

На странице проекта в Notion создай раздел ровно такой структуры:

```
## Профиль для AI-бота

**Описание**: 2–4 предложения — что за проект, для кого, на какой стадии

**Стек**: WordPress, WooCommerce, Cloudflare

**AI-направления:**
- HIGH: товарные описания, чат-консультант, рекомендации
- MEDIUM: SEO-генерация, обработка отзывов
- LOW: видеоконтент, голосовой поиск

**Keywords**: ecommerce, woocommerce, чат-бот, рекомендации, retail AI

**Anti-keywords**: crypto, NFT, autonomous driving
```

Правила:

- **Заголовок** должен быть `## Профиль для AI-бота` (heading_2). Регистр не важен, но текст должен совпадать.
- **Поля** оформляются как `**Имя**: значение` или `**Имя:** значение` — парсер примет оба варианта.
- **HIGH / MEDIUM / LOW** — список через `-`, каждый пункт начинается с `HIGH:` (регистр не важен), затем перечисление через запятую.
- **Keywords / Anti-keywords** — через запятую, без кавычек.
- Всё, что между этим заголовком и следующим `## …`, парсится. Дальше — игнорируется (там можешь писать задачи, чек-листы, что угодно — это для тебя, не для бота).

## 6. Запустить синхронизацию

```
/sync
```

В чате. Бот пройдётся по всем проектам, обновит keywords / описания / use_cases. Результат:
```
🔄 Sync: 3/4 проектов обновлено за 1240ms

Ошибки:
- bage: block_missing
```

Возможные ошибки:
- `fetch_error` — у integration нет доступа к странице. Проверь шаг 2.
- `block_missing` — не нашёлся заголовок «Профиль для AI-бота». Проверь шаг 5.
- `parse_warning` — заголовок есть, но какие-то поля не распознались. Лог покажет какие; обычно это формат с лишним/недостающим `*`.

## 7. Что бот будет делать с этим

- При scoring новостей он подставляет описание + ai_use_cases + keywords в промпт. LLM использует это чтобы решить, насколько новость релевантна каждому проекту.
- Anti-keywords ограничивают `project_score` сверху (≤ 20).
- Keywords также участвуют в **pre-filter** — дешёвом regex-фильтре, который отсеивает 60–70% мусора без вызова LLM.

Чем точнее `keywords` и `ai_use_cases` — тем меньше нерелевантных новостей в дайджесте и тем меньше расход на LLM.
